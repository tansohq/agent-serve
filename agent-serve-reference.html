Overview

Agents are becoming buyers. If they can't sign up, authenticate, pay, and use your product programmatically, they'll route spend to competitors that let them self-serve.

One command — /agent-serve — audits a SaaS product across six areas and tells the developer exactly what to build.

1. Onboarding

Can an agent create an account without a browser?

If signup requires a browser, email click, or phone number, agents can't onboard. They'll route to competitors that let them self-serve.

What to Check

  • Does a POST /signup or POST /accounts endpoint exist?
  • Does it require browser interaction (CAPTCHA, email click, phone verification)?
  • How many steps from zero to "has an account with API access"?
  • Is there a sandbox/trial mode accessible without approval?
  • Can agents provision immediately and claim later (deploy-first-claim-later)?

Gold Standards

Cloudflare (2026): Agents create accounts, buy domains, start paid subscriptions, deploy apps. Zero manual steps beyond ToS acceptance. REST/JSON catalog, identity attestation + OAuth, tokenized payment with configurable spend cap.
Stripe Projects: Agents get scoped API keys per project with explicit capabilities (read billing, create charges up to threshold). GA since April 2026 with 32 providers.

Deploy-First-Claim-Later Pattern

The most agent-friendly onboarding pattern: let agents provision immediately, require identity later.

  • Netlify: netlify deploy --allow-anonymous — 1-hour claim window
  • Prisma: npx create-db — 24-hour claim window, free permanent tier if claimed

Pattern: POST /v1/provision returns scoped credentials + TTL. No signup gate.

2. Authentication

Can an agent prove identity without human ceremony?

Machine-to-machine auth must work without human interaction. If auth requires a browser click, email link, or phone call, agents can't authenticate.

Auth Hierarchy (best to worst for agents)

  1. API keys with scopes (simplest, most agents support)
  2. OAuth Client Credentials (enterprise-grade, short-lived tokens)
  3. Service accounts with key pairs (GCP model, secure but complex)
  4. OAuth Authorization Code + PKCE (requires browser, but possible)
  5. Magic links / email OTP (agent-hostile)
  6. CAPTCHA-gated (agent-impossible)

Gold Standards

OAuth 2.0 Client Credentials: Machine-to-machine auth. No user interaction. Short-lived tokens. POST /oauth/token with grant_type=client_credentials.
Stripe API Keys: Secret key for server-side, restricted keys with granular permissions, key rotation via API, test mode keys for sandbox.
Web Bot Auth: Profile of RFC 9421 (HTTP Message Signatures) for bot authentication. Agents register public keys in a well-known directory; WAFs verify identity without CAPTCHA. Live at Cloudflare, AWS WAF, Vercel, Shopify, Akamai.

3. Purchasing

Can an agent select a plan and pay via API?

If purchasing requires a browser checkout page or "contact sales," agents can't buy. They need plan catalog + payment method + subscription creation — all via API.

The Pattern (works now, no special programs)

Most SaaS already uses Stripe. The fix is usually "expose what Stripe already supports as API endpoints."

  1. Expose GET /v1/plans returning plan IDs, prices, features, limits
  2. Accept POST /v1/subscriptions with plan_id + payment_method
  3. Accept payment_method (pm_...) — attach to customer, then subscribe. No browser.
  4. Fall back to 402 + checkout URL only when no payment method provided
  5. Use Stripe Payment Intents API (not Checkout Sessions) for programmatic payment

Manual invoice / PO processRequires humanProgrammatic billing via Stripe
Pricing hidden behind demoAgent can't evaluatePublish pricing.json

Annual contracts onlyAgents need flexibilityMonthly or usage-based option

Tokenized Payments (2026)

Saved payment methods (pm_...) — the familiar Stripe pattern. Human saves a card via Setup Intent, agent reuses it. Works today, locked to one merchant.

Shared Payment Tokens (SPTs) — time-limited, scoped to a specific business, revocable. Work cross-merchant via Link wallet.

Issuing for agents — actual virtual card numbers with per-transaction spend controls and real-time authorization.

For most SaaS: accept saved pm_... via your API first. SPTs and virtual cards will flow in from agents that already have them.

4. Usage Monitoring

Can an agent track its own consumption?

If usage data lives only in a dashboard or monthly email invoices, agents can't self-throttle, budget, or react to limits.

What to Check

  • Do API responses include rate limit headers (x-ratelimit-remaining-*, x-ratelimit-reset-*)?
  • Is there a dedicated usage API (GET /v1/usage) with current-period data?
  • Is there a billing/invoice API?
  • Can agents see quotas and how close they are to limits?
  • Are there webhooks for threshold events (80% quota, overage, billing failure)?

Gold Standards

OpenAI: Every response includes x-ratelimit-limit-requests, x-ratelimit-remaining-requests, x-ratelimit-limit-tokens, x-ratelimit-remaining-tokens, x-ratelimit-reset-*. Plus dedicated GET /v1/usage.
Twilio: Usage Records API with filtering by category, date range. Daily/monthly/all-time granularity. Includes cost per resource type.

6. Dev Readiness New

Is your API good to build against?

Access gates (signup, auth, billing) are necessary but not sufficient. If the API has vague errors, no retry safety, missing pagination, or no test mode, agents fail at integration even after they get in. Developers give up. Both route to competitors with better DX.

Good DX serves humans today and agents tomorrow. Agents have zero tolerance for ambiguity, undocumented behavior, and human-only flows. The same improvements that help agents self-correct on errors also help developers debug faster.

What to Check

Error Responses

  • Do errors return structured JSON (not HTML 500 pages)?
  • Is the format RFC 9457 Problem Details, or at minimum: type/code, message, which parameter failed?
  • Do errors include agent-actionable fields: is_retriable, retry_after_seconds, documentation_url?
  • Is there a request ID on every response (X-Request-Id) for tracing?
  • Do 429 responses include Retry-After with specific backoff?

Idempotency

  • Do mutating endpoints (POST/PATCH/DELETE) accept an Idempotency-Key header?
  • What's the dedup window? (24 hours minimum)
  • Are responses identical on replay (same status code, same body)?
  • Is there a reconciliation path (lookup-by-client-reference)?

Pagination

  • Do list endpoints paginate?
  • Is pagination cursor-based (stable across mutations) or offset-based (breaks when items change)?
  • Does the response include has_more or equivalent?

Versioning

  • Is the API versioned? Can clients pin a version (header or URL)?
  • Is there a deprecation policy with timeline (minimum 12 months)?
  • Do breaking changes get announced before they ship?

OpenAPI Spec

  • Is there a published OpenAPI spec? Is it complete?
  • Are all enum values, field constraints, and parameter formats documented?
  • Are operation IDs descriptive and action-oriented (createInvoice not postData)?

Documentation & Discoverability

  • Does /llms.txt exist with curated technical content?
  • Does /llms-full.txt exist with complete embedded documentation?
  • Do docs include runnable code examples (not pseudocode)?
  • Is there an API explorer / playground?

SDKs & Time-to-First-Call

  • Are there official SDKs? In which languages?
  • Is there a quickstart: zero to first working request in under 5 minutes?
  • How many steps from install to first successful API call?
  • Is there a CLI scaffolding tool (npx create-* pattern)?

Test Mode / Sandbox

  • Is there a test mode with separate keys (sk_test_* pattern)?
  • Are there test fixtures / magic values (test card numbers, test phone numbers)?
  • Can test environments be provisioned and torn down programmatically?
  • Is test data isolated from production?

Gold Standards

Stripe Error Responses: 
{
  "error": {
    "type": "card_error",
    "code": "card_declined",
    "decline_code": "insufficient_funds",
    "message": "Your card has insufficient funds.",
    "param": "payment_method"
  }
}

Why it works: type = category, code = machine-parseable, param = which field to fix. Every response includes Request-Id.

Stripe Idempotency: Idempotency-Key header on POST/PATCH/DELETE. 24-hour dedup window. Replay returns original response. Agents operate in unreliable environments — with idempotency, retry is always safe.
Stripe Versioning: Stripe-Version: 2024-04-10. Account pins at signup. Old versions supported for years. Agents pin a version and never break.
Cursor Pagination: GET /v1/customers?limit=10&starting_after=cus_abc123 → { "data": [...], "has_more": true }. Offset breaks when items change between pages.

Minimum Viable DX Checklist

The smallest set of changes that makes an API dev-ready for humans AND agents:

  1. Structured errors — JSON with type, code, message, param
  2. Request IDs — X-Request-Id on every response
  3. Idempotency — Idempotency-Key on POST/PATCH/DELETE
  4. Cursor pagination — starting_after + has_more
  5. API versioning — pin via header, deprecation policy
  6. OpenAPI spec — complete, versioned, published
  7. Test mode — separate keys, test fixtures, isolated
  8. Retry-After — on 429 responses with specific backoff
  9. llms.txt — curated technical content at domain root
  10. Quickstart — zero to first API call in under 5 minutes

MCP Server Design

Patterns for building agent-facing MCP servers that don't suck.

Tool Naming

Use {service}_{action}_{resource} in snake_case. Example: slack_send_message not send_message. Service prefix avoids collisions when multiple MCP servers operate together.

Tool Descriptions

Write for AI agents, not humans. "Use this when the user wants to view a customer's billing history" beats "Gets billing data." This is the single most important factor in whether agents select the right tool.

Curated Tool Surface

Anti-pattern: Auto-generating one MCP tool per API endpoint for a 200-endpoint API. Agents facing 100+ tools cannot reliably select the right one. Tool descriptions count as input tokens on every request.

A focused set of 10-15 well-designed tools beats a full API dump. Cloudflare's approach for large APIs: two tools (search() and execute()) where the model writes code against a typed representation of the API spec.

Error Handling

Report tool errors within result objects (isError: true in content), NOT as MCP protocol-level errors. This allows the LLM to see and recover from errors. Input validation errors should also be Tool Execution Errors to enable model self-correction.

Pagination

Cursor-based with has_more/next_cursor. Default 20-50 items per response. Offer both list (recent) and search (filtered) tools — agents struggle with multi-page iteration but excel at describing search criteria.

Tool Annotations

Provide readOnlyHint, destructiveHint, idempotentHint, openWorldHint. Hints for agents and permission systems to make safe decisions.

Authentication

OAuth 2.1 with PKCE is required for remote public MCP servers. For internal/personal servers, static Bearer tokens or API keys are acceptable.

MCP ↔ OpenAPI

MCP quality is capped by underlying API/OpenAPI quality. Auto-generation tools handle mechanical translation — but vague operation IDs become vague tool names, missing parameter descriptions become agent hallucinations. Good MCP servers start from good APIs with comprehensive OpenAPI specs.

Who ships MCP servers today: Stripe, Supabase, Cloudflare, Sentry, Expo, Hugging Face.

A2A Protocol

Google's Agent-to-Agent protocol. Donated to Linux Foundation in 2025.

Major supporting organizations: AWS, Cisco, Google, IBM, Microsoft, Salesforce, SAP.

Agent Card

Every A2A-compatible agent publishes a machine-readable JSON Agent Card:

{
  "name": "Billing Agent",
  "description": "Manages subscriptions and invoices",
  "provider": { "organization": "Acme Corp" },
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "skills": [
    { "name": "manage_subscription",
      "description": "Create, update, or cancel subscriptions" }
  ],
  "securitySchemes": ["apiKey", "oauth2"],
  "signature": "sha256:base64..."
}

Task Lifecycle

SUBMITTED → WORKING → COMPLETED / FAILED / CANCELED / REJECTED

Two interrupted states: INPUT_REQUIRED (needs clarification) and AUTH_REQUIRED (needs authentication). Terminal states close the task; interrupted states allow resumption.

Layering

A2A sits on top of APIs and MCP. Don't recommend A2A unless the product already has solid APIs. The progression: REST API → MCP server → A2A Agent Card.

Agent Toolkit Pattern

How to build an SDK for AI agent frameworks.

Stripe Agent Toolkit (@stripe/agent-toolkit):
  • Curated operations — not the full API surface. 10-15 operations for autonomous use.
  • Framework-native — works with OpenAI Agents SDK, Vercel AI SDK, LangChain, CrewAI. toolkit.get_tools() returns framework-compatible definitions.
  • Permission scoping — restricted API keys (rk_*) limit agent access. Read-only mode is trivial.
  • Sandbox-first — test in sandbox before production. Agent behavior is non-deterministic.

Maturity Model

Level 0: Agent-Hostile

  • Browser-only everything
  • No API
  • CAPTCHA + email verification
  • "Contact sales" for pricing

Level 1: API Exists

  • REST API for core product features
  • API keys (manually generated)
  • Docs available online
  • But: no programmatic signup, billing, or management

Level 2: Agent-Possible

  • API signup (with some friction)
  • API keys with scopes
  • Usage visible via API
  • Billing still requires dashboard

Level 3: Agent-Friendly

  • Full programmatic onboarding
  • OAuth Client Credentials
  • Plan selection + payment via API
  • Usage + billing API
  • Configuration via API

Level 4: Agent-First (Gold Standard)

  • Zero-friction agent provisioning
  • MCP server as product interface
  • Spending policies (human sets guardrails, agent executes)
  • Agent identity verification
  • Self-serve everything
  • Examples: Stripe Projects, Cloudflare (2026)

Dev Readiness: An Intersecting Dimension

The levels above measure ACCESS — can agents get in? Dev readiness measures QUALITY — is it good once they're in? These are orthogonal.

  • Level 3 access + poor dev readiness: everything is API-accessible but errors are vague, no idempotency, no test mode.
  • Level 1 access + great dev readiness: only core features exposed but the API is beautifully designed.

At Level 2, dev readiness becomes the differentiator. At Level 3-4, it's table stakes.
